# 城迹 - 软件开发计划

## 一、项目概述

### 1.1 项目名称
城迹（CityTrace）- 地理知识答题微信小程序

### 1.2 品牌理念
**极简风格，"迹"谐音"记"** —— 记录你探索世界的足迹，在答题中丈量祖国山河与全球城市。

### 1.3 项目定位
面向成年人的地理知识学习答题小程序，通过闯关游戏和随机挑战形式帮助用户熟悉祖国大好河山及全球国家城市，寓教于乐。

### 1.4 核心价值
- **教育价值**：以趣味答题形式普及地理知识
- **文化价值**：启动页推荐城市/国家文化习俗特产，拓展视野
- **社交价值**：排行榜竞争激发学习动力，分享传播知识
- **游戏价值**：星级评价、随机模式、商店系统增强趣味性

### 1.5 目标用户
- **主要用户**：18岁以上成年人
- **用户特征**：对地理知识感兴趣，喜欢轻量级知识学习，碎片化时间娱乐

## 二、核心机制说明

### 2.1 星级系统

| 属性 | 说明 |
|------|------|
| 评价单位 | 星星（可半星） |
| 满分 | 5星 |
| 完美评价 | 6星（100%正确率 + 时间达标） |
| 重复闯关 | 仅评星增加时才增加星级 |

**星级计算规则**：
```
基础星级 = 正确率对应星级
时间加成 = 答题时间达标可额外获得0.5-1星
最终星级 = 基础星级 + 时间加成（上限6星）
```

### 2.2 门票系统

| 类型 | 说明 |
|------|------|
| 闯关门票 | 最高30次，每10分钟恢复1次 |
| 随机模式门票 | 每天3次，每日0点重置 |
| 获取方式 | 商店使用星星兑换 |

### 2.3 随机模式

| 属性 | 说明 |
|------|------|
| 血量 | 初始10点，答错-1 |
| 结束条件 | 血量为0 |
| 每日次数 | 3次 |
| 题目类型 | 根据闯关层级提供相关国家题目（含国旗猜国家） |
| 奖励 | 金币（商店专属）+ 星星 |

## 三、项目阶段规划

### 3.1 整体路线图

```
Phase 1 (MVP) → Phase 2 (增强版) → Phase 3 (完整版)
   5周              3周               3周
```

### 3.2 Phase 1：最小可用版本（MVP）

**目标**：验证核心玩法，快速上线获取用户反馈

**开发周期**：5周

#### 交付内容

| 模块 | 功能点 | 优先级 |
|------|--------|--------|
| 用户系统 | 微信授权登录、用户信息存储、智能省份初始化 | P0 |
| 题库系统 | 三种题型（城市猜省份、城市猜国家、古称猜今名）、本地题目生成算法 | P0 |
| 答题系统 | 闯关答题、星级计算、闯关判定 | P0 |
| 关卡系统 | 层级解锁逻辑、区域主题解锁 | P0 |
| 门票系统 | 闯关门票、冷却恢复机制 | P0 |
| 排行榜 | 总星级排行、关卡进度排行 | P0 |
| 启动页 | 每日推荐展示、跳过/查看机制 | P1 |
| 分享功能 | 成绩分享、小程序码分享 | P1 |

#### 完成标准

- [ ] 用户可完成微信授权登录并智能初始化省份
- [ ] 三种题型可正常随机出题
- [ ] 闯关逻辑完整：答题→评星→判定→解锁
- [ ] 门票系统正常运行（30次上限，10分钟恢复）
- [ ] 排行榜数据准确展示
- [ ] 启动页功能正常运行
- [ ] 分享功能可正常唤起

#### 验收规则

1. **功能验收**
   - 所有P0功能100%可用
   - P1功能至少90%可用
   - 无阻塞性Bug

2. **性能验收**
   - 页面加载时间 < 2秒
   - 题目生成响应时间 < 300ms
   - 支持1000并发用户

3. **兼容性验收**
   - iOS微信最新版本正常运行
   - Android微信最新版本正常运行
   - 主流机型适配（iPhone 8及以上、Android 8.0及以上）

## 四、技术架构建议

### 4.1 前端架构

```
微信小程序原生开发 / Uni-app跨平台框架
├── 页面层（Pages）
│   ├── 启动页（每日推荐）
│   ├── 首页
│   ├── 关卡地图页
│   ├── 答题页
│   ├── 答题结果页
│   ├── 随机模式页
│   ├── 随机模式选择页
│   ├── 商店页
│   ├── 排行榜页
│   └── 个人中心页
├── 组件层（Components）
│   ├── 题目组件
│   ├── 星级组件
│   ├── 血量组件
│   ├── 门票组件
│   └── 分享组件
├── 业务层（Business）
│   ├── 题目生成器（本地）
│   ├── 星级计算器
│   ├── 门票管理器
│   └── 道具管理器
└── 工具层（Utils）
    ├── 请求封装
    ├── 数据缓存
    ├── 本地存储
    └── 工具函数
```

### 4.2 后端架构

```
Node.js / Python / Java
├── API服务层
│   ├── 用户接口
│   ├── 答题接口
│   ├── 排行榜接口
│   ├── 商店接口
│   └── 推荐接口
├── 业务逻辑层
│   ├── 答题逻辑
│   ├── 星级算法
│   ├── 解锁逻辑
│   ├── 门票逻辑
│   └── 推荐算法
└── 数据层
    ├── MySQL（用户数据、答题记录）
    ├── Redis（排行榜缓存、会话管理、门票状态）
    └── MongoDB（题库数据、国家数据）
```

### 4.3 本地题目生成方案

```
题目生成流程：
1. 获取关卡配置（区域、难度、题型比例）
2. 从本地/远程获取国家/城市数据
3. 根据规则生成题目
   - 城市猜省份：随机选择城市 + 生成干扰项
   - 城市猜国家：随机选择城市 + 生成干扰项
   - 古称猜今名：随机选择古称 + 生成干扰项
   - 国旗猜国家：随机选择国旗 + 生成干扰项
4. 缓存已生成题目，避免重复
```

### 4.4 技术选型建议

| 类型 | 推荐方案 | 理由 |
|------|----------|------|
| 小程序框架 | 原生开发 / Uni-app | 原生性能最优，Uni-app可跨平台 |
| 后端语言 | Node.js / Python | 开发效率高，生态丰富 |
| 数据库 | MySQL + Redis + MongoDB | 关系数据+缓存+文档数据各取所长 |
| 云服务 | 腾讯云 / 阿里云 | 国内访问稳定，微信生态友好 |
| 部署方式 | Docker容器化 | 便于扩展和维护 |
