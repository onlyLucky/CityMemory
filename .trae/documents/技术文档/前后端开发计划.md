# 城迹项目 - 前后端开发计划

## 一、文档概述

### 1.1 文档目的
本文档定义"城迹"项目的详细前后端开发计划,包括开发阶段划分、任务分配、时间规划、技术实现、资源需求、风险评估、协作流程、联调计划、测试策略和交付标准,为开发团队提供明确的执行指南。

### 1.2 项目概述

**项目名称**: 城迹 - 地理知识问答微信小程序

**项目目标**: 构建一款地理知识问答类微信小程序,用户通过答题闯关的方式学习地理知识

**技术架构**:
- 小程序端: UniApp + Vue 3 + Pinia + TypeScript
- 后台管理: Vue 3 + Element Plus + TypeScript
- 服务端: Node.js + Koa + TypeScript (统一服务端)
- 数据库: MySQL + Redis + MongoDB

**项目周期**: 22周 (约5.5个月)

### 1.3 团队配置

|------|------|----------|

#### 5.1.3 网络请求封装

**技术方案**:
```typescript
// utils/request.ts
import type { RequestOptions } from '@dcloudio/uni-app'

const BASE_URL = import.meta.env.VITE_API_BASE_URL || 'https://api.citytrace.com/api/v1'

interface RequestConfig extends RequestOptions {
  showLoading?: boolean
  showError?: boolean
}

class Request {
  private baseURL: string

  constructor() {
    this.baseURL = BASE_URL
  }

  private getHeader(): Record<string, string> {
    const token = uni.getStorageSync('token')
    const headers: Record<string, string> = {
      'Content-Type': 'application/json',
      'X-Platform': 'miniprogram'
    }
    if (token) {
      headers['Authorization'] = `Bearer ${token}`
    }
    return headers
  }

  async request<T>(options: RequestConfig): Promise<T> {
    const {
      url,
      method = 'GET',
      data,
      showLoading = true,
      showError = true
    } = options

    if (showLoading) {
      uni.showLoading({ title: '加载中', mask: true })
    }

    return new Promise((resolve, reject) => {
      uni.request({
        url: this.baseURL + url,
        method,
        data,
        header: this.getHeader(),
        success: (res: any) => {
          if (showLoading) {
            uni.hideLoading()
          }

          const { code, message, data: responseData } = res.data

          if (code === 0) {
            resolve(responseData)
          } else if (code === 1002) {
            uni.removeStorageSync('token')
            uni.reLaunch({ url: '/pages/auth/index' })
            reject(new Error(message))
          } else {
            if (showError) {
              uni.showToast({ title: message || '请求失败', icon: 'none' })
            }
            reject(new Error(message))
          }
        },
        fail: (error) => {
          if (showLoading) {
            uni.hideLoading()
          }
          if (showError) {
            uni.showToast({ title: '网络错误', icon: 'none' })
          }
          reject(error)
        }
      })
    })
  }

  get<T>(url: string, data?: unknown): Promise<T> {
    return this.request<T>({ url, method: 'GET', data })
  }

  post<T>(url: string, data?: unknown): Promise<T> {
    return this.request<T>({ url, method: 'POST', data })
  }

  put<T>(url: string, data?: unknown): Promise<T> {
    return this.request<T>({ url, method: 'PUT', data })
  }

  delete<T>(url: string, data?: unknown): Promise<T> {
    return this.request<T>({ url, method: 'DELETE', data })
  }
}

const request = new Request()

export default request
```

**实现要点**:
- 封装uni.request方法
- 统一处理请求头、loading、错误提示
- 自动添加token
- 处理token过期情况
- 提供get、post、put、delete方法

#### 5.1.4 关卡系统实现

**技术方案**:
```typescript
// composables/useLevel.ts
import { ref, computed } from 'vue'
import { levelApi } from '@/api/level'
import type { Level, LevelProgress } from '@/types'

export function useLevel() {
  const levels = ref<Level[]>([])
  const currentLevel = ref<Level | null>(null)
  const progress = ref<LevelProgress | null>(null)
  const loading = ref(false)

  const fetchLevels = async (province?: string) => {
    loading.value = true
    try {
      levels.value = await levelApi.getLevelList({ province })
    } finally {
      loading.value = false
    }
  }

  const startLevel = async (levelId: string) => {
    loading.value = true
    try {
      const result = await levelApi.startLevel(levelId)
      currentLevel.value = result.level
      progress.value = result.progress
      return result
    } finally {
      loading.value = false
    }
  }

  const submitAnswer = async (levelId: string, questionId: string, answer: string) => {
    loading.value = true
    try {
      const result = await levelApi.submitAnswer(levelId, questionId, answer)
      if (progress.value) {
        progress.value.blood = result.blood
        progress.value.currentQuestionIndex++
      }
      return result
    } finally {
      loading.value = false
    }
  }

  const completeLevel = async (levelId: string, stars: number, blood: number, answers: any[]) => {
    loading.value = true
    try {
      const result = await levelApi.completeLevel(levelId, stars, blood, answers)
      await fetchLevels()
      return result
    } finally {
      loading.value = false
    }
  }

  const isLevelUnlocked = (level: Level) => {
    if (level.levelOrder === 1) return true
    const previousLevel = levels.value.find(l => l.levelOrder === level.levelOrder - 1)
    return previousLevel?.isCompleted || false
  }

  return {
    levels,
    currentLevel,
    progress,
    loading,
    fetchLevels,
    startLevel,
    submitAnswer,
    completeLevel,
    isLevelUnlocked
  }
}
```

// utils/jwt.ts
import jwt from 'jsonwebtoken'

const JWT_SECRET = process.env.JWT_SECRET || 'your-secret-key'
const JWT_EXPIRES_IN = '7d'

export interface JwtPayload {
  id: string
  role?: string
}

export const jwtUtil = {
  sign(payload: JwtPayload): string {
    return jwt.sign(payload, JWT_SECRET, {
      expiresIn: JWT_EXPIRES_IN
    })
  },

  verify(token: string): JwtPayload {
    try {
      return jwt.verify(token, JWT_SECRET) as JwtPayload
    } catch (error) {
      throw new Error('Token无效或已过期')
    }
  },

  decode(token: string): JwtPayload | null {
    try {
      return jwt.decode(token) as JwtPayload
    } catch (error) {
      return null
    }
  }
}

import { jwtUtil } from '@/utils/jwt'
import { error } from '@/utils/response'
import { ErrorCode } from '@/constants/error'

export const authMiddleware = async (ctx: Context, next: Next) => {
  try {
    const token = ctx.headers.authorization?.replace('Bearer ', '')

    if (!token) {
      error(ctx, { code: ErrorCode.UNAUTHORIZED, message: '未提供认证令牌' })

}

      return
    }

## 七、风险评估及应对措施

### 7.1 技术风险

#### 7.1.1 技术选型风险

**风险描述**: 选用的技术栈不成熟或不适合项目需求

**风险等级**: 中

**影响**: 项目延期、质量下降

**应对措施**:
1. 充分调研技术栈的成熟度和适用性
2. 进行技术选型评审
3. 准备备选方案
4. 加强技术培训

**责任人**: 技术负责人

#### 7.1.2 性能风险

**风险描述**: 系统性能不达标,无法满足高并发需求

**风险等级**: 中

**影响**: 用户体验差、系统崩溃

**应对措施**:
1. 提前进行性能测试
2. 制定性能优化方案
3. 使用缓存、负载均衡等技术
4. 进行压力测试

**责任人**: 后端开发、运维工程师

#### 7.1.3 安全风险

**风险描述**: 系统存在安全漏洞,数据泄露或被攻击

**风险等级**: 高

**影响**: 数据泄露、系统瘫痪

**应对措施**:
1. 进行安全审计
2. 实施安全防护措施
3. 定期进行安全测试
4. 建立应急响应机制

**责任人**: 后端开发、运维工程师

### 7.2 业务风险

**风险描述**: 需求频繁变更,导致开发进度延误

**风险等级**: 高

**影响**: 项目延期、成本增加

## 八、前后端协作流程

### 8.1 协作原则

1. **接口先行**: 后端先定义接口,前端根据接口开发
2. **文档同步**: 接口文档实时更新,保持同步
3. **及时沟通**: 遇到问题及时沟通,快速解决
4. **版本控制**: 使用Git进行版本控制,规范提交
5. **代码审查**: 代码提交前进行审查,保证质量

### 8.2 接口定义流程

```
需求分析 → 接口设计 → 接口评审 → 接口文档 → 接口开发 → 接口测试 → 接口交付
```

**步骤说明**:

1. **需求分析**: 根据需求文档,分析需要哪些接口
2. **接口设计**: 设计接口的URL、方法、参数、响应
3. **接口评审**: 前后端一起评审接口设计
4. **接口文档**: 编写详细的接口文档
5. **接口开发**: 后端开发接口
6. **接口测试**: 后端测试接口功能
7. **接口交付**: 接口交付给前端使用

### 8.3 前后端协作模式

#### 8.3.1 并行开发模式

**适用场景**: 功能模块相对独立

**协作方式**:
- 后端先定义接口文档
- 前端根据接口文档开发
- 后端开发接口实现
- 前后端联调

**优点**:
- 开发效率高
- 可以并行开发

**缺点**:
- 接口变更影响大
- 需要良好的文档

#### 8.3.2 串行开发模式

**适用场景**: 功能模块依赖性强

**协作方式**:
- 后端先开发接口
- 接口测试通过后交付前端
- 前端根据接口开发
- 前后端联调

**优点**:
- 接口稳定
- 减少返工

**缺点**:
- 开发效率低
- 不能并行开发

### 8.4 接口文档管理

**工具选择**: Swagger / YApi / Apifox

**文档内容**:
- 接口URL
- 请求方法
- 请求参数
- 响应格式
- 错误码
- 示例代码

**文档维护**:
- 接口变更及时更新文档
- 定期审查文档准确性
- 保持文档与代码同步

### 8.5 沟通机制

#### 8.5.1 日常沟通

**工具**: 微信、钉钉、Slack

**频率**: 随时

**内容**:
- 接口问题

**时间**: 每周五下午15:00

**内容**:

**时间**: 接口开发前

**参与人**: 前端开发、后端开发、项目经理

**内容**:
- 接口设计评审
- 参数讨论

### 8.6 代码协作

#### 8.6.1 分支管理策略

**分支类型**:
- master: 主分支,用于生产环境
- develop: 开发分支,用于集成开发
- feature/*: 功能分支,用于开发新功能
- hotfix/*: 修复分支,用于紧急修复

**分支流程**:
```
master ← develop ← feature/*
         ↑
      hotfix/*
```

#### 8.6.2 提交规范

**提交格式**:
```
<type>(<scope>): <subject>

<body>

<footer>
```

**类型(type)**:
- feat: 新功能
- fix: 修复bug
- docs: 文档更新
- style: 代码格式
- refactor: 重构
- test: 测试
- chore: 构建/工具

**示例**:
```
feat(user): 添加用户登录功能

实现微信登录和JWT认证

Closes #123
```

#### 8.6.3 代码审查

**审查流程**:
1. 开发者提交Pull Request
2. 审查者审查代码
3. 提出修改意见
4. 开发者修改代码
5. 审查通过,合并代码

**审查要点**:
- 代码规范性
- 逻辑正确性
- 性能优化
- 安全性
- 测试覆盖
